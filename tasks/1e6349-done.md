# Add the daily “Floor Is Lava” event

## Context and existing architecture

The app is a browser-only ES-module tic-tac-toe game under `public/`:

- `public/js/player.js` owns the normalized, persisted `tic-tac-toe-player` localStorage record, local-date helpers, daily-gift revision logic, player rewards, and `pending_coins`.
- `public/js/controller.js` coordinates a first-to-three match (`MATCH_POINTS_TO_WIN = 3`), normal results/rewards, the existing home coin-presentation queue, and storage-event synchronization.
- `public/js/view.js`, `public/index.html`, and scoped rules in `public/css/styles.css` own the screens, dialogs, home widgets, focus, and animation/reduced-motion behavior.
- `public/js/leaderboard.js` already provides deterministic seeded pseudo-random utilities and simulated competitors.
- The game uses Paper.css. Keep custom CSS scoped to `.game` and necessary for the game UI; do not globally override Paper.css.

Implement a polished, casual daily event named **Floor Is Lava**. It is a separate event flow that reuses normal computer tic-tac-toe gameplay and its normal result accounting, subject to the event-specific rules below.

## Daily time and persistence rules

1. Use the player's device **local calendar day and local clock**. The event is open from **06:00 inclusive through before 18:00** each local day.
2. Persist all event state in the existing player localStorage record and normalize/migrate malformed or missing event data safely. Reconcile/reset it whenever the record is loaded or time is evaluated:
   - reset to a fresh attempt on a new local date;
   - retain that date's state during the same date, including after reload;
   - retain a closed partial (0–5 win) attempt read-only after 18:00 until the next local-day reset.
3. Persist at least an event date, state/status (active, eliminated, completed/claimed, and closed partial as applicable), consecutive event win count, a monotonic revision/version, and a one-time pending home-progress-celebration indicator. Design the persisted shape so invalid values cannot mint rewards or resurrect a completed/eliminated attempt.
4. Handle browser `storage` events like the existing player synchronization: all tabs must reconcile to the authoritative daily attempt. Render the event/status again after a storage update where appropriate.
5. Treat the shared stored attempt as authoritative across tabs. Before accepting an event match result, compare/re-read the expected date, revision, current event state, and stage. Commit exactly one valid result. A stale competing tab must discard its result **entirely**—no event progress, normal game statistics, leaderboard/battle-pass point, ordinary match coins, or pot payout—and reconcile to storage.
6. Loading/reloading/closing a tab during an unfinished event match abandons that board without progress, elimination, or rewards. It may later start the same unplayed stage again. Do not persist an in-progress board.
7. A match validly started before 18:00 may finish after the cutoff and still count, including a sixth win: it can complete the event and pay the pot. Never allow starting a new event match at/after 18:00.

## Deterministic daily field and reward

1. Build a deterministic **date-only** simulated field shared by all players on a given local calendar day (not personalized). Follow the leaderboard's deterministic simulation approach; expose pure/testable helpers for the event seed/field/schedule rather than burying it in DOM code.
2. Generate exactly **12** stable daily opponents, with stable readable names and lightweight deterministic illustrated/SVG-style portraits. No profile/avatar-customization feature is required.
3. Seed a shuffled daily order. The player faces the first six opponents, one per event stage; the remaining six still appear on the climb.
4. Seed **5–10** of the 12 as simulated event finishers. They are the other winners whenever the local player completes the run. Their seeded names, portraits, winner set, positions, and schedule must remain stable after reload that date.
5. Simulate the field over the open window from current local time, analogous to the leaderboard: bots jump at deterministic checkpoints among six discrete island levels; designated finishers reach the reward island by 18:00 and non-finishers stop below it. Refresh/re-render while the event screen is visible so time-based position changes are visible. Do not use continuous movement.
6. Six **completed first-to-three matches** are required. Event progress is isolated from ordinary matches: non-event games may be played between event games and never alter this event streak.
7. A drawn event match is replayed with no progress change and no elimination. A completed event match lost by the player eliminates the player for that local day.
8. On the sixth event win, award once and immediately persist a share of the 100-coin pot: `floor(100 / (1 + simulated finisher count))`. The player plus the 5–10 seeded simulated finishers are the winners, and any remainder is unawarded. Keep the exact potential share hidden before completion; reveal the exact earned share at the summit/completed state.
9. Each event match is still a normal match for existing accounting: board/round statistics and final match win/loss/streak behavior, a final local match win's leaderboard point and battle-pass point, and the existing 3-coin or 4-coin streak award all apply. Add the one-time pot payout only on the sixth event win. Send all coins through the existing durable `coin_balance`/`pending_coins` flow.

## Home launcher and event screen

1. Add a Floor Is Lava launcher below the Daily Gifts launcher in the existing left-side `.home-widget-list--left` stack. Give it a clear accessible name and a simple inline SVG island icon. It remains visible in every daily state and opens the event screen rather than being disabled or hidden.
2. Add a dedicated secondary Floor Is Lava screen, not a dialog. It must include the app's established Back treatment; Back returns to Home. Do not add an in-match Back/abandon control—the active game keeps its current no-back UI.
3. The screen must fit entirely in its design canvas with **no internal scrolling**. Make the layout compact and responsive while preserving readable controls, names/portraits, and keyboard focus.
4. Make the event feel lively and casual: show the 12 opponent portraits and a clearly highlighted local `You` portrait/token progressing from bottom to top over six island levels. Use a large reward island/100-coin pot at the top. Animate discrete portrait jumps and player advancement when motion is permitted; keep the player journey prominent even with all 12 bots visible.
5. Add concise live-region feedback for meaningful changes (stage/result/status) and preserve sensible focus behavior: focus the screen heading on entry, the actionable next match/status control as appropriate, and the launcher on Back.
6. Screen states/actions:
   - **Before 06:00:** show an attractive “opens at 6:00 AM” status/countdown and the field; no start action.
   - **Open active attempt:** show the 100-coin pot, climb, current stage/progress, and a prominent action to start the next named daily opponent.
   - **Eliminated:** show a friendly “come back tomorrow” outcome; no start action.
   - **Completed:** show the summit/winner summary and exact floored payout, with no duplicate claim or replay.
   - **After 18:00 with unfinished attempt:** show a clear closed-for-today / return-tomorrow status with the earned partial climb retained read-only.
7. The active match should display the selected named daily opponent and portrait in the normal match UI while retaining the existing random-AI move behavior. Ordinary games continue to use the generic Computer opponent.
8. Event result handling differs from ordinary result handling: after any event result, Continue returns to the Floor Is Lava screen. A win/draw shows the next-stage/retry situation; a loss/final win shows its defeated/completed status. Normal Continue keeps returning Home.

## Home progress celebration

1. After each successfully committed event match win (stages 1–5 only), persist a one-time pending progress celebration. Do not create it for draws, loss, completion, closed/out-of-window state, or a stale multi-tab result.
2. When the player next returns to Home (including after a reload), only while the same event is still open, active, and unfinished, play a Mario-like celebration at the Floor Is Lava launcher: the button pops and a nearby `+1` appears. Clear/consume the flag once shown so it cannot replay.
3. Integrate this presentation with the existing coin animation queue rather than competing with it. Match its safeguards: queue/order it safely with pending coin presentations, cancel and clean up timers/DOM on navigation or screen changes, visibility/lifecycle changes, and repeated invocation, and do not leave stale effects behind. Ensure completion/reload/storage reconciliation cannot cause duplicate celebration.
4. Respect `prefers-reduced-motion`: preserve the information/state change and accessible announcement but use immediate, non-jumping/non-popping state updates for the climb, launcher, and all new decorative animation.

## Suggested implementation boundaries

- Add a focused event/domain module for date seed, roster, finisher selection, discrete schedule/positions, window/status calculation, validation, and payout computation. Keep it DOM-free and deterministic with injectable timestamps as existing leaderboard code does.
- Extend `player.js` normalization and persistence with event-state functions that atomically verify and commit a stage result/reward using the existing daily-gift compare/re-read pattern as a model. Keep currency and durable state changes out of the view.
- Extend `GameController` with event navigation, event-match context, cutoff/start validation, result commit-before-normal-reward handling, result routing, home celebration scheduling, timers, and storage reconciliation. Do not allow the ordinary `recordResult` path to award stale event results before validation.
- Extend HTML/View/CSS with the launcher, no-scroll event screen, accessible SVG portraits/islands, status/countdown rendering, named event opponent, and motion-safe animations. Keep normal game, leaderboard, gifts, profile, battle pass, and style flows working.

## Tests and acceptance criteria

Add/extend Vitest coverage (using deterministic clocks/storage/randomness) for at least:

1. Event-window boundaries (before 06:00, 06:00, before 18:00, 18:00), local-date reset, closed partial state, and a pre-close match finishing validly after 18:00.
2. Date-only deterministic 12-opponent roster/order/portraits, 5–10 finishers, all-player consistency for one day, changed field on a different day, six first opponents, discrete time-based bot positions, and finisher/non-finisher destinations.
3. Event persistence/normalization, reload abandonment, six-win progression, draw retry, loss elimination, one-time completion, and exact floor-shared 100-pot payout with no remainder award.
4. Event opponents and normal Computer opponents, existing normal match stats/leaderboard/battle-pass/streak coin rewards, and event result Continue routing.
5. Cross-tab/storage revision races: only one result and all its rewards commit; stale results receive none and tabs reconcile.
6. Launcher placement/accessibility below Daily Gifts, all event screen states and Back/focus behavior, no event starts while unavailable, and no internal event-screen scroll.
7. Persisted one-time launcher `+1` celebration, valid playback after returning/reloading Home, interaction with coin presentation, cancellation/cleanup, invalid-state suppression, and reduced-motion behavior.
8. Existing test suite remains green; add CSS/DOM tests where the project convention makes visual structural requirements testable.

Manual acceptance: during a local 06:00–18:00 window, a player can repeatedly enter the island event, play named seeded opponents in normal first-to-three tic-tac-toe, see a lively compact bottom-to-top 12-portrait climb, survive draws, be eliminated on a match loss, or receive normal six-match benefits plus exactly one calculated shared pot after six wins. The home launcher visibly celebrates each unfinished-stage win on the next Home return without duplicate rewards or motion when reduced motion is requested.
