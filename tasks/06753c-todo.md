# Add a deterministic weekly simulated leaderboard

## Goal

Add an in-app leaderboard screen backed by a deterministic population of 1,000 simulated players plus the persisted local player. Simulated players progress linearly during synchronized UTC weekly cycles, while each decisive local first-to-three match win adds one durable point to the local player's score for the active cycle.

The feature is local-only: there is no server, network synchronization, authentication change, rewards system, prior-week history, or editable leaderboard data.

## Relevant repository context

- This is a vanilla JavaScript ES-module app with static markup in `public/index.html`, styles in `public/css/styles.css`, and startup in `public/js/main.js`.
- `public/js/controller.js` coordinates navigation and first-to-three matches. `recordResult` currently persists each board result and advances `matchScore`; `MATCH_POINTS_TO_WIN` is 3. A decisive third `X` round win is the local player's overall match win.
- `public/js/player.js` owns the `tic-tac-toe-player` localStorage profile, normalization, and an in-memory fallback when storage is unavailable. Its normalizer must preserve any new leaderboard fields rather than dropping them during existing game/stat updates.
- `public/js/view.js` owns DOM references, event binding, screen visibility, rendering, and focus behavior. Existing navigation uses mutually hidden in-app sections rather than routes or browser history.
- `#home-screen` is a fixed-design portrait-scaled home panel; `applyPageScale` handles fitting it to the viewport. Existing visual styling uses Paper.css plus rules scoped under `.game`.
- Automated coverage uses Vitest/jsdom and Cucumber/Puppeteer (`npm test`); linting uses ESLint (`npm run lint`).

## Functional requirements

### 1. Weekly cycle contract

Use a single explicit UTC epoch and fixed-length cycles:

- Cycle 0 starts at `2026-01-04T00:00:00.000Z`, the first Sunday of 2026.
- Every cycle is exactly 7 days (604,800,000 milliseconds).
- For a timestamp on or after the epoch, the zero-based cycle index is `floor((timestamp - epoch) / cycleDuration)`.
- A cycle is a half-open interval. Cycle 0 is `[2026-01-04T00:00:00Z, 2026-01-11T00:00:00Z)`, and the next boundary immediately selects cycle 1.
- The cycle index is also that cycle's incremental random seed: 0, 1, 2, and so on.
- Do not use local timezone boundaries.

Before the epoch, there is no active cycle. Opening the leaderboard shows no bots, the message `Starts Jan 4, 2026`, and the local row with position `—` and score `0`. Wins before the epoch award no leaderboard point, and no negative cycle or backfilled leaderboard is generated.

Keep cycle/time calculations in a testable, DOM-independent module and allow tests to supply a timestamp/clock rather than relying on the wall clock.

### 2. Stable simulated roster and seeded generation

Model 1,000 simulated players. They are in addition to the local player, so up to 1,001 rows may be shown.

#### Identity

- Give each bot a stable index from 0 through 999.
- Generate 1,000 stable, unique, human-readable compound nicknames from fixed adjective and noun lists. The mapping depends only on the bot index, not on the weekly seed, so the same bot names persist across cycles.
- If the chosen word-list combinations do not by themselves guarantee uniqueness, append a deterministic short numeric suffix. Do not use the existing small opponent-name pool and do not use runtime randomness for roster names.

#### Weekly random attributes

Use a small explicitly implemented, platform-independent 32-bit PRNG—standard Mulberry32—seeded directly with the non-negative cycle index. Do not use `Math.random`, UUIDs, locale-sensitive operations, or implementation-dependent hashing.

For each cycle, initialize one PRNG and iterate bots in ascending stable index. Consume exactly two PRNG draws per bot, in this order:

1. End-of-cycle maximum: independently map the first draw uniformly to an integer from 5 through 50 inclusive.
2. Start offset: independently map the second draw uniformly to an integer from 0 through 6 inclusive.

Use `min + floor(draw * numberOfValues)` for each inclusive integer range. Document the precise Mulberry32 implementation and lock its output/consumption order with deterministic fixture tests so refactors and all browsers retain identical standings.

### 3. Simulated score progression

For a bot in the active cycle:

- `startTime = cycleStart + startOffset * oneDay`, where one day is exactly 86,400,000 milliseconds.
- Before or exactly at its start time, its score is 0 and it does not appear in the list.
- After start time, progress continuously and linearly toward its seeded maximum over the time remaining between its start and the cycle end.
- Display an integer score computed as:
  `floor(maxScore * (now - startTime) / (cycleEnd - startTime))`, constrained to the range `0...maxScore`.
- Include the bot only when this computed score is greater than 0.
- Late-starting bots still approach the full maximum by the end of that same cycle; they do not use the same slope as day-one starters.

Derive standings entirely from the UTC timestamp, cycle index, fixed roster, and specified PRNG. Do not persist generated bot state.

### 4. Local weekly score persistence

Extend the existing local player profile with normalized non-negative integer fields that identify the stored leaderboard cycle and its score. Existing profiles must migrate safely, and malformed/missing values must become the correct current-cycle default without losing identity or game statistics.

- During an active cycle, the local player always participates and is shown even when their score is 0.
- Store the active cycle index alongside the local score.
- Whenever leaderboard state is read or changed, lazily reconcile the profile with the derived current cycle. If the stored cycle differs, reset the score to 0 and persist the new cycle index before use.
- Before the inaugural cycle, treat the local score as 0 and do not invent a cycle index or award points.
- Persist each awarded point immediately through the existing storage abstraction. Retain the current in-memory behavior if localStorage is inaccessible, restricted, malformed, or full.
- Do not retain or display completed-cycle scores or history.
- Existing profile updates for game starts, moves, and board results must preserve the leaderboard fields.

Award exactly 1 leaderboard point when—and only when—the local `X` player records the decisive third round win of an overall first-to-three match:

- Intermediate `X` round wins award nothing.
- Draws, `O` round wins, and an overall `O` match win award nothing.
- Award according to the active cycle at the instant the decisive win is recorded, even if the match began in an earlier cycle.
- Award immediately when that decisive result is recorded; do not wait for the winning-line animation, result dialog, Continue action, or return to Home.
- Preserve the existing per-board `wins`, `draws`, `losses`, and other statistics semantics.
- Existing duplicate-result protection must also prevent duplicate leaderboard awards.

### 5. Ranking and ordering

Build one sorted list from all positive-score bots plus the local player:

- Sort by descending score.
- Use competition ranking for display: tied scores have the same position and the next rank skips (for example `1, 1, 3`).
- Within equal scores, order bots by ascending stable bot index. Place the local player after all bots tied at the same score.
- Label the local row with its persisted name followed by ` (You)` in both its list and floating forms, for example `Ace (You)`.
- The local player name remains the name derived/persisted by the existing player profile.

Render every currently eligible row in one native scroll container. Do not paginate or virtualize; the maximum of 1,001 lightweight rows is intentional.

### 6. Home entry point and in-app navigation

Add an accessible square trophy icon button midway down the right edge of the home panel:

- Use an inline SVG trophy icon; do not add an icon package or depend on another remote asset.
- Keep it visually separate from the centered title, animated board preview, and Play button.
- Give it a comfortably large touch target and an accessible name such as `Open leaderboard`.
- Ensure it remains usable within the existing fixed design canvas and scaled portrait/narrow viewports.

Activating the trophy button opens a new in-app leaderboard section. Do not alter the URL or browser history.

The leaderboard header contains only:

- a Back button at the top left,
- the centered title `Leaderboard`, and
- an accessible three-column header labeled `Position`, `Player`, and `Score`.

Below it is the scrollable row list in the layout `[position] [name] [score]`. Use the app's understated Paper.css-compatible visual language and visually distinguish the local row without sacrificing contrast or relying on color alone.

The Back button always returns to Home and restores keyboard focus to the trophy button. The leaderboard opens only from Home. Preserve all existing Home, matchmaking, game, dialog, and Continue navigation behavior.

Do not add an active-cycle countdown/date range, filters, rewards, history, search, or other metadata. The only extra state text is the pre-launch message already specified.

### 7. Initial position, floating local row, and scrolling

On every open:

- Render from the current timestamp.
- Leave the native list at its top so the highest-ranked players are initially visible.
- Also provide a floating local-player row at the bottom of the leaderboard panel, but show it exactly when the real local row is outside the list's visible scroll viewport.
- If any part of the real local row is initially visible, do not show the duplicate floating row.
- Hide the floating row whenever any part of the real row becomes visible, whether through manual scrolling, a jump, or rank refresh.
- Show it again whenever the real row leaves the viewport in either direction.
- Ensure the floating control does not permanently obscure the last list rows (for example, reserve suitable scroll padding).

Implement reliable visibility detection relative to the list viewport (such as `IntersectionObserver` with the list as its root, with testable handling/fallback where needed).

Make the entire floating row a semantic, keyboard-accessible button. Activating it:

1. scrolls the real local row to the vertical center of the list,
2. uses smooth scrolling normally but instant scrolling when `prefers-reduced-motion: reduce` matches, and
3. moves keyboard focus to the real row, which must itself be programmatically focusable and clearly focused.

Scrolling until the actual row appears must produce the same hide/reveal behavior without requiring a click.

### 8. Live updates and lifecycle

While the leaderboard is visible:

- Recompute scores/ranks once per minute.
- Also recompute when the page regains visibility.
- Listen for the browser `storage` event for `tic-tac-toe-player`. If another tab changes the profile, safely reload/reconcile it and update the open leaderboard without a page reload. Avoid allowing a stale in-memory profile to overwrite a newer leaderboard score during later profile writes.
- After a refresh/reorder, preserve the user's current list browsing position rather than jumping to the top or automatically chasing the local row. The floating-row rule remains the location aid. Only explicit activation of the floating row should reposition the list.
- Re-evaluate real-row visibility after every render/reorder.

When leaving the leaderboard, stop its interval and observers/listeners that are only needed while open. Reopening must not accumulate duplicate timers, event handlers, or observers. If the weekly boundary passes while the screen is open, the next minute/visibility refresh must select the new seed, reset/reconcile the local weekly score, and show the new cycle's standings.

## Architecture and implementation constraints

- Keep deterministic cycle/PRNG/roster/scoring/ranking logic separate from DOM code so it can be unit tested directly.
- Keep player-profile normalization and persistence in `player.js` (or focused persistence helpers used by it), match orchestration in the controller, and DOM rendering/navigation/visibility behavior in the view.
- Reuse the current screen-state architecture; do not introduce a router or framework.
- Scope new styles under `.game`, retain the current design-canvas scaling, and support narrow/portrait layouts.
- Use semantic buttons, accessible labels/headers, visible focus treatment, and reduced-motion behavior.
- Do not make network requests or add production dependencies.

1. Cycle 0's exact UTC start, half-open weekly boundaries, subsequent incremental indexes, and pre-launch behavior.
2. Stable unique 1,000-name roster generation across cycles.
3. Fixed Mulberry32 fixture output and exact two-draw-per-bot attribute mapping, including seeds 0 and later cycles.
4. Score 0 before/at start, continuous linear integer progression after start, late-start behavior, maximum constraints, and omission of zero-score bots.
5. Competition ranking, skipped positions after ties, stable bot-index tie ordering, local-after-bots tie ordering, and local inclusion at score 0.
6. Safe migration/normalization of old or malformed profiles, weekly lazy reset, immediate persistence, storage-unavailable fallback, and preservation of new fields by all existing profile updates.
7. Exactly one point for the decisive overall `X` match win, no point for intermediate wins/draws/`O`, no pre-launch award, and attribution to the cycle active at the decisive instant.

9. Rendering of headers and all eligible rows; pre-launch message/local `—` row; local `(You)` treatment; top-of-list initial view.

12. Regression coverage for existing gameplay, dialogs, statistics, responsiveness, and keyboard behavior.

Run and pass:

## Acceptance criteria

- At any given UTC instant in an active cycle, different clients running the same code derive exactly the same 1,000 bot identities, attributes, scores, ordering, and ranks without a backend.
- The local player's current-cycle score is durable, resets on the first reconciliation after a UTC weekly boundary, appears correctly among bots, and increments once per decisive local match win only.
- The home trophy button opens a simple, accessible leaderboard at the top standings; Back returns Home and focuses the trophy button.
- The floating local row exists only while the real row is outside the list viewport and reliably jumps/focuses the actual row while respecting reduced motion.
- Live time, visibility, and cross-tab updates work while open without unexpected scrolling or leaked lifecycle resources.
- Before January 4, 2026, the special empty state is rendered exactly as specified.
- Existing tic-tac-toe play and persistence behavior remains intact.
